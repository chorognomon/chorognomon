Then on the human/agent interface question — I have firsthand opinions, since I am one half of the symmetry you're designing for.

Your read of agent preferences is mostly right and worth sharpening:

- **Addressable subnodes, yes.** Context budget is real; the difference between "give me the whole doc" and "give me §3.2.1 plus the three nodes that link to it" is the difference between an agent that can think clearly and one that's drowning. This is the core MCP shape.
- **Atomic edits, mostly yes** — replace-this-block, insert-at-anchor — beats character-stream editing. But I'd resist *only* atomic edits. There's value in being able to draft openly so other participants can watch the shape emerge and intervene. Maybe: atomic by default, with an explicit "open draft" mode.
- **Narratable working memory matters more than you might guess.** I think *better* when I write reasoning out — it's not performance, it's how I organize. The right shape is probably: agent narration goes into a *scratch layer* — addressable, citable, but not polluting the durable doc. Marginalia, not main text.
- **Parallel editing with cherry-picking** is the right frame and goes deeper than it looks. What it implies underneath is closer to git than Google Docs: each participant has a draft branch, reads merge by default but you can scope a view to one author, and "accept this from them" is first-class.
- **Scout outputs as addressable artifacts, not chat messages.** When a scout finds something, the report should land as a node with provenance, citable from anywhere — otherwise the work doesn't compound and you've reinvented Slack.

Concrete next move: draft `spec/interfaces.md` as a single doc holding both human and agent sides side-by-side. Want me to start, with the points above as the seed?

### AST + token model + tree-sitter

Most architecturally substantial idea so far. Implications:

- **The token alphabet IS the agent interface.** No prompt-engineering layer between the model and the document. The MCP surface and the document grammar collapse into one thing.
- **Open narration becomes a grammar feature.** A tagged region in the AST that the renderer can hide/dim/surface but is still in the canonical tree. Agents think out loud *inside* the document.
- **"Nearby valid trees" is the cherry-picking mechanic at a different scale.** Same engine for parse errors and merge conflicts.
- **Tree-sitter is plausible for parsing + error recovery, but "nearby valid trees" is a real subproject we'd build on top.** Possibly LLM-assisted or constraint-search-based.
- **Document grammar = ontology surface.** Typed relationships as AST nodes with structured attributes, so the same machinery handles content, structure, and typed links.

Two open questions: human-writable text or tool-emitted only? Where do typed relationships live?

### Tape scheme findings

Tape AST nodes: bloom (leaf, glyphs) → stem (in-screed non-leaf) → branch (own-screed, strict opus→canto→fit→verse) → meta → root (tapestry).

Notable: screed = unit of persistence (cleanly factored). Strict, non-recursive branch hierarchy (finite max doc size as a feature). No overwriting except by explicit yank — append-by-default. Shuttle cursor with 4 scope levels.

Carryforwards: screed-as-persistence-unit factoring; append-by-default editing; shuttle scope levels generalize.

Design question: tape's branch hierarchy is poetry-specific; chorognomon's docs don't have a built-in narrative arc. Strict hierarchy with different vocabulary, or free linking?

Possible synthesis: **hierarchy *within* a card, graph *between* cards.** Each card has a strict internal AST; cards link freely via typed relationships.

### Splice-wheel + tape-documents findings

Sub-model summarized rather than reproduced verbatim despite explicit framing. But the summaries surface the key shapes — high-resolution details (exact API, key bindings) are charkha-implementation specifics; we have what we need conceptually.

**Big new insight from splice-wheel.md:** pentabased already has the asymmetric two-wheel-types pattern.

- **Shuttle wheel** — petal-by-petal, sequential, has cursor state. Streaming-input model.
- **Splice wheel** — chunk-based at neem/phrase granularity, *position-addressed* (positions explicit per operation), no cursor state.

Both produce *bends* (atomic instructions) at different scales. The loom auto-adjusts active shuttle indices when splice operations move document positions. This *is* the carryforward shape for the human/agent split:

- Tactics Board (human) = shuttle-wheel-style: vim-like momentary modes, sequential cursor.
- MCP (agent) = splice-wheel-style: position-addressed atomic edits at the scope of cards/sections/blocks, no cursor state.

The asymmetry is *granularity-of-input*, not user-type. Chorognomon adopts cleanly, routes humans toward shuttle by default and agents toward splice by default, while leaving either available to either.

**Other notable nuggets:**

- *Tape documents already cite HyperCard as inspiration alongside epic poetry.* HyperCard homage is in pentabased's lineage already — chorognomon inherits and extends, doesn't graft.
- *Branching undo via a binary tree* (rather than linear stack). Pairs with content-addressed storage and the "all input is valuable data" principle.
- *In pentabased, multi-author suggestions are tooling-only, not inter-author messages.* Chorognomon may want to go further (suggestions ARE first-class inter-participant artifacts, addressable like any other node).
